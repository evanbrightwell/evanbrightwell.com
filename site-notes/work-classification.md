# Work Classification Pass

Date: 2026-06-23

This is an editorial classification pass for the website. It is not a full archive.
The goal is to keep the strongest work visible and prevent category drift.

## Category Rules

## Image-Level Sorting Model

Behance projects, local folders, and future uploads are treated as source
archives, not final website categories.

The website should classify at the image level first, then group selected images
back into visitor-facing projects. One source project can feed multiple website
buckets when the images prove different things.

Example: a single Behance project may contain a room render, a tabletop tray, a
kit detail, and a process image. Those should not all inherit the same website
category. The room can go to Spaces, the tray can go to Display Systems, the kit
detail can go to Product & Packaging, and process imagery can go to Process /
Systems.

Every candidate image should carry:

- Source project: where it came from.
- Website bucket: where it belongs on the site.
- Visual role: hero, supporting, detail, proof, or archive.
- Strength score: whether it improves the site compared with current work.
- Replacement note: what older image or project it can remove if stronger.
- Exclusion reason: cover image, weak crop, wrong bucket, older version, detail
  only, duplicate, or off-strategy.

Future ingestion should work as a ranked replacement pass:

1. Add new work into a source folder.
2. Generate contact sheets from all source images, excluding Behance album
   covers always.
3. Classify each image by visual role, not source project.
4. Compare against the current curated set.
5. Keep only images that strengthen a category, complete a project story, or
   reveal a capability not already represented.
6. Demote or remove older images when new work is stronger.

This means the site should eventually be driven by a curated image manifest, not
by raw Behance project groupings.

### Spaces

Full environments and spatial proposals: rooms, lounges, bars, dinners, event
worlds, outdoor activations, and hospitality takeovers.

Use this category when the image proves spatial thinking: layout, scale,
lighting, atmosphere, brand world, guest path, or client-ready environmental
visualization.

Bars can stay in Spaces, but most one-off bar renders should be grouped into a
single `Bars` project instead of treated as separate major projects.

Do not use Behance album cover images for selection. They are treated as rough
album packaging, not editorial image choices. This is a hard exclusion, not a
"avoid as lead" preference.

### Display Systems

Mid-scale physical brand systems: glorifiers, carts, trays, bottle displays,
retail/on-premise fixtures, tabletop systems, and sculptural presentation
objects.

Use this category when the image proves product presence in physical form. This
is the bridge between Spaces and Product & Packaging.

Recommended section label: `Display Systems`, not just `Displays`.

### Product & Packaging

Object-forward product imagery: kits, packaging, bottles, launch compositions,
serveware, gifting systems, and product-led compositions.

Use this category when the object, package, kit, or launch asset is the main
thing being evaluated.

### Process / Systems

Workflow-forward work: procedural systems, visual tools, motion, sculpting,
custom pipelines, experiments, and selected older work that supports the idea
that Evan can build new approaches.

Use this category sparingly. It should differentiate the practice, not become a
dumping ground.
